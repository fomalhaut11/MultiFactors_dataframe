## 🎯 如何使用说明文档

### 阅读时机

- **入门阶段**: 先学教程，再读概念说明加深理解
- **遇到困惑**: 操作时不理解为什么，查看说明文档
- **系统学习**: 想要全面理解系统设计，系统阅读架构文档
- **提升阶段**: 想要写出更好的代码，学习最佳实践

### 阅读建议

- **可以跳读**: 根据需要选择感兴趣的部分
- **结合实践**: 边读边在代码中验证
- **思考对比**: 理解设计选择背后的权衡
- **记录笔记**: 记录自己的理解和疑问

## 📖 文档类型对比

| 类型 | 说明文档 (Explanation) | 教程 (Tutorial) |
|------|------------------------|-----------------|
| 目标 | 理解概念和原理 | 学习技能 |
| 结构 | 主题式组织 | 步骤式组织 |
| 内容 | 为什么 (Why) | 怎么做 (How) |
| 适用场景 | 加深理解 | 上手实践 |

**举例说明**:
- **说明文档**: "为什么要使用TTM而不是单季度数据" - 解释原因和背景
- **教程**: "如何计算TTM数据" - 手把手教操作

## 📐 说明文档写作规范

### 文档结构

```markdown
# 主题标题

**创建日期**: YYYY-MM-DD
**最后更新**: YYYY-MM-DD
**适用版本**: vX.X.X

## 概述

用1-2段话说明这个主题是什么

## 背景

为什么需要了解这个主题？它解决什么问题？

## 核心概念

详细解释相关概念

### 概念1

解释概念1

### 概念2

解释概念2

## 工作原理

说明系统如何实现这些概念

## 设计权衡

为什么这样设计？有哪些权衡考虑？

## 常见误解

澄清常见的误解和错误认知

## 深入阅读

- [相关概念](...)
- [学术论文](...)
```

### 写作原则

1. **清晰易懂**: 用简单的语言解释复杂的概念
2. **逻辑连贯**: 概念之间的关系要清晰
3. **举例说明**: 用具体例子帮助理解
4. **图文并茂**: 适当使用图表辅助说明
5. **避免说教**: 说明而不是命令

## 💡 概念说明 vs 操作指南

### 什么时候写概念说明？

- 解释"为什么"而不是"怎么做"
- 说明背景和原理
- 澄清术语和概念
- 解释设计决策

### 什么时候写操作指南？

- 解决具体问题
- 提供操作步骤
- 快速完成任务
- 不需要深入理解

### 两者关系

```
概念说明 ← 互相补充 → 操作指南
   ↓                      ↓
理解原理              完成任务
   ↓                      ↓
为什么这样做        具体怎么做
```

## 💡 贡献说明文档

欢迎贡献新的说明文档！请遵循以下原则：

1. **理解导向**: 目标是帮助读者理解概念
2. **清晰表达**: 用简单的语言解释复杂的事物
3. **逻辑结构**: 概念之间要有清晰的逻辑关系
4. **举例说明**: 用具体例子帮助理解抽象概念
5. **避免命令**: 说明而不是指令

说明文档模板：

```markdown
# 主题名称

## 什么是XXX

用简单的语言解释这个概念

## 为什么需要XXX

说明这个概念解决什么问题

## XXX如何工作

解释工作原理和机制

## 常见误解

澄清常见的错误理解

## 相关概念

- [相关概念1](...)
- [相关概念2](...)

## 深入阅读

- 学术论文或参考资料
```
